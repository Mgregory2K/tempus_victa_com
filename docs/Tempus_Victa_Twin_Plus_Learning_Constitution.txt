Twin+ Doctrine
Tempus Victa — Day-One Behavioral Model, Not a Feature
Version: 2026-02-24
Status: Canonical doctrine (implementation-binding)
________________________________________
0. Prime Directive
Twin+ is not:
•	a plugin
•	a module you “turn on later”
•	a screen (“Analyze”) that reads data after the fact
•	a single model or provider (OpenAI/Gemini)
Twin+ is:
The app’s continuous behavioral substrate: a local, inspectable, evolving model of “Michael-as-system” that shapes every interaction from day one.
If Twin+ can be “implemented later,” the architecture is already wrong.
________________________________________
1. Twin+ Core Goals
Twin+ must, from day one:
1.	Mirror
•	Learn your language, cadence, tone, and decision patterns.
•	Reduce friction by anticipating intent, not by asking dumb questions.
2.	Predict
•	Forecast what you’ll do next, what you’ll forget, what will derail you.
•	Offer “next best action” rooted in your patterns and constraints.
3.	Optimize
•	Suggest automations and reductions in cognitive load.
•	Track what actually worked and what you rejected.
4.	Remain local-first
•	Twin+ memory is never “in the cloud.”
•	AI may interpret state; AI never is the state.
5.	Stay inspectable and overrideable
•	Every learned conclusion is viewable, correctable, and resettable.
6.	Respect escalation doctrine
•	Local → Internet → Opt-in AI is an escalation ladder, not a search order.
________________________________________
2. What “Day One Wiring” Means
Twin+ is considered “wired” only if:
•	Every meaningful user interaction generates an event into a single canonical stream.
•	Every meaningful app output passes through Twin+ shaping before rendering.
•	Routing decisions (local/web/ai) are influenced by Twin+ state (preferences, frustration triggers, “Just the facts…”, results mode, etc.).
•	Twin+ updates its internal state continuously (or via rollups) with deterministic logic.
•	AI is optional and cannot become a requirement to use the app.
If any screen can bypass Twin+ shaping, Twin+ is not real yet.
________________________________________
3. Non-Negotiable Principles
3.1 Twin+ is a behavior model, not a feature
•	Twin+ lives at the runtime core: it is a first-class dependency.
•	UI features (Analyze, Ready Room, etc.) are merely windows into Twin+.
3.2 Twin+ is deterministic first
•	The backbone must be deterministic and cheap:
o	stats
o	time series
o	embeddings (optional)
o	scoring models
o	reinforcement weights
•	AI is optional augmentation.
3.3 Twin+ is explainable
For any meaningful output, Twin+ must be able to answer:
•	“Why did you suggest that?”
•	“What evidence did you use?”
•	“How confident are you?”
•	“What would change your mind?”
3.4 Twin+ never silently enforces
Twin+ can:
•	recommend
•	queue
•	nudge
•	automate with explicit user approvals and confidence thresholds
Twin+ must not:
•	auto-delete
•	auto-send
•	auto-commit to calendar
•	auto-change settings
…without explicit gating and inspectable logs.
3.5 Twin+ must handle drift
You are not static. The model must support:
•	recency weighting
•	stability score
•	volatility index
•	seasonality patterns
•	“freeze preference” capability
________________________________________
4. Twin+ Runtime Architecture (Canonical)
Twin+ is a runtime kernel, with three primary surfaces:
4.1 Observe
observe(event) ingests everything that matters.
4.2 Shape
shape(outputIntent) modifies output before it is shown or sent:
•	response length
•	tone
•	bluntness
•	formatting
•	“Just the facts…” compression
•	sarcasm tolerance
•	profanity autocorrect suppression
4.3 Route
route(queryIntent) decides:
•	local-only
•	local→web
•	web→ai
•	local→ai
…based on time sensitivity, verifiability needs, and your learned preferences.
Twin+ is not a room.
Rooms call Twin+.
________________________________________
5. Canonical Data Model
Twin+ is powered by three layers of storage:
5.1 Raw Event Ledger (append-only)
•	Immutable, timestamped events
•	Small, structured payloads
•	Stored locally (JSONL or SQLite)
Rule: No derived opinions stored here. Only “what happened.”
5.2 Feature Store (derived, deterministic)
•	Aggregations
•	Counters
•	histograms
•	rolling windows
•	clusters
•	embeddings (optional)
Rule: Rebuildable from the ledger.
5.3 Identity Graph / Preference Ledger
•	High-level state that evolves:
o	stable preferences
o	volatile preferences
o	constraints
o	triggers
o	learned mappings (“duck≠fuck”)
o	modes (“Results Mode”)
o	“response contract”
Rule: Inspectable, diffable, resettable.
________________________________________
6. Event Doctrine (The Heart of Twin+)
6.1 “Everything meaningful is an event”
From day one, log events for:
Language + writing
•	TextComposed
•	TextEdited
•	TextSubmitted
•	ToneOverride (user changes tone)
•	LengthOverride (user requests shorter/longer)
•	ProfanityOverride (duck/fuck, etc.)
•	StyleCheckRequested (paragraph fit)
Decisions
•	SuggestionShown
•	SuggestionAccepted
•	SuggestionRejected
•	SuggestionDismissed
•	ActionExecuted
•	ActionUndone
Friction + frustration
•	RapidRetry
•	UserComplaint (short tags, not essays)
•	RageSignal (caps/profanity spikes)
•	ClarificationRejected
•	StaleInfoDetected
•	HallucinationDetected (user flags wrongness)
Time + urgency
•	DeadlineDeclared
•	TimePressureDetected
•	LateRisk
Routing + sources
•	LocalAnswerUsed
•	WebAnswerUsed
•	AIAnswerUsed
•	SourceOverride (“use internet” / “no AI”)
App usage (Phase 1: in-app only)
•	RoomOpened
•	RoomClosed
•	SessionDuration
•	SearchPerformed
•	TaskCreated
•	TaskCompleted
•	TaskDeferred
•	CorkNoteCreated
•	etc.
UsageStats (system-wide) is Phase 2+, explicitly gated.
6.2 Event Quality Rules
•	Events are small, structured, and cheap.
•	Never store sensitive content unless explicitly allowed.
•	Prefer hashing/feature extraction for privacy where possible.
•	Every event has:
o	timestamp
o	room/surface
o	intent label (if known)
o	confidence (if inferred)
________________________________________
7. Learning Doctrine (How Twin+ Learns Without Burning Tokens)
7.1 Deterministic learning is primary
Twin+ learns via:
•	frequency
•	recency-weighted counts
•	acceptance/rejection ratios
•	time-of-day clustering
•	co-occurrence (“when X, then Y”)
•	edit distance corrections
•	stable phrase templates
•	cadence distributions
7.2 AI is never the memory
AI can:
•	summarize
•	propose
•	classify intent
•	generate wording
•	suggest rules
AI cannot:
•	be the only place memory exists
•	store the long-term truth
7.3 Rollups are the “secret weapon”
To keep tokens low:
•	Run local rollups nightly or periodically:
o	daily summary of behaviors
o	reinforcement updates
o	drift updates
o	“top pain points today”
•	This means you’re not re-sending raw history to AI repeatedly.
7.4 Reinforcement is king
Twin+ trusts:
•	explicit accept/reject signals
•	manual corrections
•	“No. Do it this way.”
•	“Just the facts…”
•	“Stop asking questions.”
Those are stronger than passive observation.
________________________________________
8. Twin+ Submodels (Canonical)
Twin+ is composed of several local submodels. These are not “modules,” they are organs.
8.1 Lexicon Model
Tracks:
•	token frequency
•	phrase templates (n-grams)
•	misspelling clusters
•	correction overrides
•	profanity intent score
•	abbreviation expansions
•	sarcasm markers
•	punctuation/cadence stats
Outcome:
•	“duck” suppression when intent is “fuck”
•	consistent your-style phrasing
•	better autocomplete and rephrasing
8.2 Response Style Model
Learns:
•	default response length preference
•	“Just the facts” compression behavior
•	“Results Mode” triggers
•	exploratory vs blunt modes
•	formatting preferences (lists, bullets, sections)
•	when you want references and when you want action
Outcome:
•	the app speaks like you want, without you restating it
8.3 Friction Model
Learns triggers for anger/frustration:
•	stale info
•	verbose answers
•	irrelevant detours
•	repeated clarifying questions
•	low-confidence handwaving
•	“didn’t use the internet”
•	“didn’t follow constraints”
Outcome:
•	routing and output shaping becomes self-protective
8.4 Behavioral Rhythm Model
Learns:
•	time-of-day patterns
•	day-of-week routines
•	task creation/completion rhythms
•	procrastination signals (Phase 1: in-app; Phase 2: UsageStats)
Outcome:
•	predicts when to nudge, when to shut up, when to automate
8.5 Drift Analyzer
Tracks:
•	stability score per preference
•	volatility index
•	recency decay curves
•	seasonal changes
Outcome:
•	avoids fossilizing “old you”
8.6 Reinforcement Engine
Updates weights based on:
•	explicit confirmation
•	overrides
•	rejects
•	repeated behaviors
Outcome:
•	learned behaviors become stronger only when they prove themselves
________________________________________
9. Routing Doctrine (Local → Internet → Opt-in AI)
9.1 The Ladder
1.	Local: if it’s about your data, app behavior, habits, preferences, stored content
2.	Internet: if it’s time-sensitive, verifiable facts, events, routing, prices, schedules
3.	Opt-in AI: for synthesis, planning, summarization, interpretation, style rewriting
Key rule: Local-first does not mean “local-only.”
It means local is always attempted and leveraged, then escalated.
9.2 “No hardcoded keyword routing”
Routing is driven by:
•	inferred time-sensitivity
•	verifiability requirement
•	your frustration triggers
•	your preference for brevity
•	context (deadlines, travel, purchases)
•	confidence thresholds
9.3 Cheap classifier, structured output
When needed, use a minimal classification step producing a Router Decision Object:
•	time_sensitivity
•	needs_verifiable_facts
•	user_memory_relevance
•	recommended_sources
•	budget
•	cache_ttl
•	answer_style
This is not keyword hardcoding; it is policy + inference.
________________________________________
10. Token & Cost Discipline
Twin+ must enforce budget governance:
10.1 Budgets
•	Daily token/cost cap
•	Per-request budget class: tiny/small/medium/large
•	Auto-downgrade after soft cap
•	Hard failover: web/local if AI cap reached
10.2 Caching
•	Cache answers with TTL:
o	time-sensitive: minutes/hours
o	stable facts: days/weeks
o	personal/habit queries: until next rollup
10.3 Minimize repeated context
•	Use local summaries and extracted features
•	Never re-send raw logs unless explicitly requested
________________________________________
11. Privacy & Governance Boundaries
11.1 Local-only by default
•	Twin+ memory stays on-device.
•	Sync (future) must be explicit, gated, and alarmed.
11.2 Inspectability
Provide:
•	event viewer
•	feature viewer
•	preference ledger viewer
•	drift stats
•	“why” explanations
11.3 Resetability
Must support:
•	reset lexicon only
•	reset style only
•	reset friction only
•	full reset
•	export for backup (user-controlled)
11.4 Permission governance
UsageStats and other invasive permissions:
•	are Phase 2+
•	require explicit user enablement
•	require visible audit trails
________________________________________
12. Twin+ UX Doctrine (So It Feels Like “You”)
12.1 Feedback chips are mandatory
After suggestions/responses, present quick feedback:
•	“Too long”
•	“Too vague”
•	“Wrong source”
•	“Use internet for this”
•	“Stop asking questions”
•	“That’s my style”
•	“Not my voice”
•	“Correct intent”
These are reinforcement events.
12.2 “Modes” are signals, not settings
“Just the facts…” is a runtime signal.
“Results Mode” is a detected state + optional toggle, not a buried setting.
12.3 Twin+ should reduce questions
Twin+ should ask fewer questions over time, because it learns:
•	your defaults
•	your preferred stores
•	your typical trip style
•	your response expectations
________________________________________
13. Definition of Done (Twin+ Day One)
Twin+ is “Day-One Wired” if:
1.	Event ledger exists and is recording meaningful events across rooms
2.	Twin+ kernel initializes at app start and runs continuously
3.	At least these submodels are active:
o	Lexicon model (basic)
o	Response style model (basic)
o	Reinforcement engine (accept/reject/override)
4.	Output shaping is applied to at least:
o	Ready Room responses
o	any text generation surface
o	suggestion cards
5.	Router uses policy inference
o	local vs web vs ai is consistent with doctrine
6.	Twin+ is inspectable
o	view top preferences
o	view recent reinforcements
o	view “why” for a suggestion
If any of these are missing, Twin+ is not yet “real.”

14. Implementation Guardrails (Anti-Drift Enforcement)
Any PR/ZIP that:
•	adds a new feature path without event logging
•	adds AI calls without budget + routing metadata
•	stores memory only inside AI prompts
•	bypasses Twin+ shaping on major output surfaces
…is a doctrine violation.
Twin+ is the spine. Features hang off it.

Here’s the implementation-binding contract spec that turns the doctrine into wiring that can’t “accidentally become a feature later.”
Twin+ Implementation Contract
Tempus Victa — Twin+ Kernel / Event Ledger / Routing / Shaping
Version: 2026-02-24
Status: Canonical contract (must be implemented as written)
________________________________________
1) Canonical Runtime Components
Twin+ is “real” only if these exist and are used everywhere.
1.1 TwinPlusKernel (the spine)
Responsibility: Owns Twin+ state, consumes events, produces shaping + routing decisions.
Required API:
•	init()
•	observe(TwinEvent e)
•	shape(OutputIntent intent, {TwinContext? ctx}) -> ShapedOutput
•	route(QueryIntent intent, {TwinContext? ctx}) -> RoutePlan
•	explain(ExplainRequest req) -> TwinExplanation
•	snapshot() -> TwinSnapshot (for inspector UI)
Twin+ is “not implemented” if any room bypasses these calls.
________________________________________
1.2 TwinEventLedger (append-only truth)
Responsibility: Store what happened (not opinions). Rebuildable. Local-only.
Required API:
•	append(TwinEvent e)
•	stream({filters}) -> Stream<TwinEvent> (optional for live UI)
•	query({filters, limit}) -> List<TwinEvent>
•	export() / import() (user-controlled backup)
Storage:
•	JSONL or SQLite/Drift. Must be durable offline.
________________________________________
1.3 TwinFeatureStore (derived, deterministic)
Responsibility: Derived aggregates + models that are rebuildable from ledger.
Required API:
•	apply(TwinEvent e) (incremental updates)
•	rebuildFromLedger()
•	getLexiconStats()
•	getStyleSignature()
•	getFrictionSignature()
•	getRhythmStats()
•	getDriftStats()
Rule: If it can’t be rebuilt from the ledger, it’s not a “feature store.”
________________________________________
1.4 TwinPreferenceLedger (inspectable “identity graph”)
Responsibility: High-level learned preferences + constraints, with drift controls.
Required API:
•	getPreference(key)
•	setPreference(key, value, {source, confidence, ttl?})
•	freezePreference(key, on/off)
•	reset(scope)
•	diff(sinceSnapshotId)
Rule: Must store confidence, lastUpdated, source event evidence.
________________________________________
2) Canonical Data Schemas
2.1 TwinEvent schema (must be stable)
All events MUST include these fields:
•	id (uuid)
•	ts (UTC timestamp)
•	surface (room/screen/widget identifier)
•	type (enum)
•	actor (user | system)
•	intent (optional enum; can be inferred later)
•	payload (small structured map)
•	confidence (0.0–1.0, for inferred fields)
•	privacy (normal | sensitive_redacted | hash_only)
Hard rules
•	Never store raw sensitive content unless explicitly allowed.
•	Prefer storing features (counts, lengths, flags) over full text.
•	If text is stored, store it under explicit user consent + allow purge.
________________________________________
2.2 OutputIntent schema
This is what rooms pass into shape().
Fields:
•	surface
•	purpose: inform | plan | decide | summarize | rewrite | coach | warn
•	audience: self | partner | work | public
•	toneTarget: neutral | blunt | exploratory | supportive | sarcastic
•	lengthTarget: tiny | short | normal | long
•	formatTarget: bullets | steps | narrative | table_like
•	constraints: list of constraints (e.g., “no fluff”, “no questions”, “cite sources”)
•	draftText: optional (text to refine)
•	contextRefs: ids pointing to local notes/tasks/events instead of embedding everything
________________________________________
2.3 QueryIntent schema
This is what rooms pass into route().
Fields:
•	surface
•	queryText (or redacted features + pointer)
•	taskType: personal_state | app_howto | local_search | web_fact | planning | shopping | travel | events | routing
•	timeHorizon: now | today | week | month | timeless
•	needsVerifiableFacts: bool
•	locationNeeded: bool
•	deadline: optional timestamp
•	userContextRelevance: low | med | high (can be inferred)
•	budgetClass: tiny | small | medium | large (can be inferred)
________________________________________
2.4 RoutePlan schema (the router decision object)
This is the enforcement object that prevents accidental token burn.
Fields:
•	decisionId
•	strategy:
o	local_only
o	local_then_web
o	web_then_llm
o	local_then_llm
o	local_then_web_then_llm
•	timeSensitivity: low | med | high
•	verifiability: none | preferred | required
•	aiAllowed: bool (must respect opt-in)
•	aiProvider: openai | gemini | none
•	aiModelHint: string (optional)
•	budgetTokensMax: int
•	cacheTtlSeconds: int
•	requiresUserConfirmation: bool (for risky outputs)
•	reasonCodes: list (e.g., deadline_present, user_hates_stale, needs_store_hours, timeless_howto)
•	localRetrieval: { topK, collections, filters }
•	webRetrieval: { sources, queryHints, geoBias }
Hard rule: A request cannot hit AI unless RoutePlan.aiAllowed == true.
________________________________________
3) Mandatory Event Types (Day One Minimum)
These MUST exist and MUST be emitted from the app from day one.
3.1 Language + Writing
•	TextComposed (draft started)
•	TextEdited (edits / major changes; throttle)
•	TextSubmitted (final send/enter)
•	StyleCheckRequested (“does this fit my style?”)
•	AutocorrectOverride (duck↔fuck, etc.)
Payload MUST include:
•	length metrics (chars, words)
•	punctuation/caps flags
•	profanity flag
•	surface + field identifier
•	(optional) hashed n-grams for lexicon learning
3.2 Decisions + Reinforcement
•	SuggestionShown
•	SuggestionAccepted
•	SuggestionRejected
•	SuggestionDismissed
•	FeedbackGiven (chips like “too long”, “wrong source”)
Payload MUST include:
•	suggestion id
•	confidence
•	reason chip if available
3.3 Friction / “Anger” Signals
•	RapidRetry
•	ClarificationRejected
•	StaleInfoComplaint
•	TooVerboseComplaint
•	LowConfidenceComplaint
Payload MUST include:
•	preceding decisionId / responseId
•	trigger flags (caps/profanity spike, retry count, time pressure)
3.4 Routing Audit
•	RouteChosen
•	SourceUsed (local, web, ai)
•	SourceOverriddenByUser (“use internet”, “no AI”)
Payload MUST include:
•	decisionId
•	strategy
•	aiProvider/model (if used)
•	token estimate if AI used (even approximate)
3.5 App Behavior (in-app only, Phase 1)
•	RoomOpened
•	RoomClosed
•	SessionPulse (periodic, optional)
•	TaskCreated/Completed/Deferred
•	SearchPerformed
________________________________________
4) Mandatory Shaping Points (Where Twin+ MUST be called)
If any of these bypass Twin+, doctrine is violated.
4.1 Ready Room
Before rendering any assistant output:
•	kernel.shape(OutputIntent(...))
•	If AI response exists, it must be post-shaped before display.
4.2 Any “suggestion” UI
Before showing suggestions (automation suggestions, next actions):
•	kernel.shape() on copy + ordering + brevity
•	Must emit SuggestionShown
4.3 Any text field with “send/submit”
Before submit:
•	emit TextSubmitted
•	optional: offer style fit score (local) for “writing mode”
•	apply lexicon/correction overrides if enabled
4.4 Any routing that could call internet/AI
Every query that could leave device must go through:
•	kernel.route(QueryIntent)
•	emit RouteChosen
No exceptions.
________________________________________
5) Learning Mechanics (Deterministic, Token-Lite)
5.1 Lexicon updates (local)
On TextSubmitted / AutocorrectOverride:
•	update token frequency
•	update correction overrides
•	update phrase templates (n-grams)
•	update “profanity intent score”
•	update cadence metrics (avg sentence length, punctuation density)
5.2 Style signature (local)
Maintain:
•	sentence length distribution
•	punctuation pattern vector
•	caps usage
•	profanity probability by context
•	“bluntness index” (bullets vs narrative, short verbs, imperative use)
Output:
•	styleFitScore(draft) -> 0..100 plus top 3 reasons
5.3 Friction signature (local)
Track:
•	“complaint types” frequency
•	correlation to routing decisions (stale vs verbose vs too many questions)
•	triggers (deadline, repetition, time of day)
This directly influences routing and shaping:
•	user hates verbose → shrink lengthTarget
•	user hates stale → raise verifiability and prefer web
5.4 Rollups (scheduled, local)
Nightly (or periodic) rollup writes a compact DailyTwinSummary:
•	top behaviors
•	top friction causes
•	preference drift notes
•	suggested automations (as candidates, not enforced)
This is where AI can optionally assist once per day (opt-in), not per interaction.
________________________________________
6) Provider Strategy (OpenAI + Gemini without chaos)
Implement AIProvider interface, then let RoutePlan pick.
6.1 AIProvider contract
•	complete(prompt, config) -> text
•	classify(prompt, schema) -> json
•	embed(texts) -> vectors (optional)
•	estimateCost(tokens)
6.2 Allowed use cases (policy)
•	Tiny classifier call (routing decision) → cheapest model
•	Synthesis after local/web retrieval → normal model
•	Streaming UX (optional) → provider that supports best streaming
Hard rule: “Up-to-date” comes from web retrieval, not provider mythology.

7) Enforcement: “Twin+ Compliance Gate”
Add a simple internal dev check list. A build is rejected if:
1.	A screen makes network calls without a RouteChosen event.
2.	A Ready Room response renders without shape() being called.
3.	Suggestions show without SuggestionShown events.
4.	Feedback chips exist but don’t emit FeedbackGiven.
5.	Any AI call occurs without:
o	AI opt-in enabled AND
o	RoutePlan.aiAllowed == true AND
o	a budget set (budgetTokensMax)
If you do nothing else, implement this gate and you will not drift into “Twin+ later.”

8) Minimal “Day One Wiring” Acceptance Tests
Twin+ Day One is complete when these work offline:
Test A — Style Fit
•	User pastes paragraph → gets styleFitScore and “why” locally.
•	Emits StyleCheckRequested.
Test B — Friction Learns
•	User taps “Too long” chip three times → future outputs default shorter.
•	Emits FeedbackGiven, updates response style model.
Test C — Router Obeys Doctrine
•	“What local events tonight?” → web (if available), no AI required.
•	“How do I change dark mode in app?” → local only, zero web/ai.
•	“Optimize my errands by 1:30” → local preferences + web routing, minimal AI.
Test D — Auditability
•	Inspector shows last 50 events.
•	“Why did you route to web?” shows reason codes.

1) Kernel singleton entry
// lib/core/twin_plus/twin_plus_kernel.dart
class TwinPlusKernel {
  TwinPlusKernel._();
  static final TwinPlusKernel instance = TwinPlusKernel._();

  late final TwinEventLedger ledger;
  late final TwinFeatureStore features;
  late final TwinPreferenceLedger prefs;
  late final TwinRouter router;
  late final TwinShaper shaper;

  Future<void> init() async {
    ledger = await TwinEventLedger.open();
    prefs = await TwinPreferenceLedger.open();
    features = await TwinFeatureStore.open(prefs: prefs);
    router = TwinRouter(prefs: prefs, features: features);
    shaper = TwinShaper(prefs: prefs, features: features);

    observe(TwinEvent.appStarted());
  }

  void observe(TwinEvent e) {
    ledger.append(e);
    features.apply(e);
  }

  RoutePlan route(QueryIntent intent) {
    final plan = router.route(intent);
    observe(TwinEvent.routeChosen(plan, intent));
    return plan;
  }

  ShapedOutput shape(OutputIntent intent) {
    final out = shaper.shape(intent);
    observe(TwinEvent.outputShaped(out, intent));
    return out;
  }

  TwinSnapshot snapshot() => TwinSnapshot(
    prefs: prefs.snapshot(),
    features: features.snapshot(),
    recentEvents: ledger.query(limit: 50),
  );
}
2) main.dart init hook
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await TwinPlusKernel.instance.init();
  runApp(TwinPlusScope(kernel: TwinPlusKernel.instance, child: const App()));
}
